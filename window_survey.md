# 流计算中窗口聚合算子的实现机制、学术演进与工业界系统架构对比调研报告

## 1. 窗口聚合算子的核心概念与效率瓶颈分析

在流计算系统中，由于输入的数据流具有无限性和无界性，系统必须通过窗口（Window）机制将无界的数据流切分为有限的数据块，以便进行状态计算和实时分析。一个标准的聚合算子在逻辑上由窗口分配器和聚合函数（Aggregation Function）共同定义。窗口分配器负责确定事件所属的范围，而聚合函数则定义了对落在窗口内的数据进行何种物理聚合计算。

窗口分配器中最典型且具挑战性的是滑动窗口（Sliding Window，在 SQL 中通常称为 Hop Window）。滑动窗口由窗口大小 $W$（Window Size）和滑动步长 $S$（Window Advance 或 Slide）这两个时间或计数参数来定义。当滑动步长 $S$ 小于窗口大小 $W$ 时，相邻的窗口之间会产生物理上的重叠。重叠度或重叠因子（Overlap Factor）可以表示为：

$$O = \frac{W}{S}$$

这意味着，在任何给定的时刻，系统内同时处于活跃状态的逻辑窗口数量为：

$$\mathcal{N} = \left\lceil \frac{W}{S} \right\rceil$$

对于任意进入系统的流式事件，由于窗口的重叠性，它在逻辑上属于 $\mathcal{N}$ 个不同的活跃窗口。例如，一个窗口大小为 $10$ 分钟、滑动步长为 $1$ 分钟的滑动窗口，其重叠因子 $O = 10$，意味着每一个到达的事件都必须被分配到 $10$ 个不同的窗口中进行计算。

在传统流处理系统（如 Apache Flink DataStream API 或 Apache Beam）的早期及默认实现中，这种逻辑上的重叠直接导致了物理上的数据复制与状态膨胀。为了维护窗口的独立语义，系统会将每个到来的流事件物理复制（Copy）并分发到对应的 $\mathcal{N}$ 个窗口状态实例（或称为 Pane，窗格）中。在这种"按窗格独立复制"的幼稚实现下，系统面临极大的效率瓶颈。随着滑动步长 $S$ 的减小或窗口大小 $W$ 的增加，内存中重复存储的数据量将呈线性甚至指数级上升。例如，当面对高吞吐数据流，若需要计算过去 1 小时内每秒更新一次的滑动指标（即 $W = 3600$ 秒，$S = 1$ 秒），一个事件将被复制 $3600$ 份。这不仅会消耗极其恐怖的内存资源，还会导致频繁的状态写入与极高的 CPU 反序列化开销。此外，长寿的窗口对象会导致垃圾回收（GC）进入老年代，显著拉长垃圾回收停顿时间，造成计算延迟剧烈波动。

为了缓解此问题，诸如 Flink 引入了基于 `ReduceFunction` 或 `AggregateFunction` 的增量计算机制。增量聚合不再在窗口内部缓存所有原始事件，而是只保留一个状态累加器（Accumulator），新事件到来时立即触发累加动作并更新该累加器。然而，即便采用了增量聚合，系统仍然必须为这 $\mathcal{N}$ 个相互重叠的窗口各自维护和更新一份独立的状态累加器，没有在物理层面彻底消除重叠区域的冗余计算与状态存储。因此，如何从根本上优化重叠窗口的管理，成为学术界与工业界共同关注的焦点。

## 2. 窗口管理机制与聚合函数的代数属性耦合关系

窗口聚合算子的物理实现效率、状态管理方式以及优化空间，并不是孤立决定的，而是与聚合函数（Func）的数学与代数属性（Algebraic Properties）高度耦合的。聚合函数的代数特性决定了数据是否可以被安全地拆解、压缩以及如何高效地执行回撤。

### 结合律（Associativity）与预聚合（Pre-Aggregation）

一个聚合函数 $f$ 满足结合律，是指对于任意两个不相交的数据子集 $A$ 和 $B$（其中 $D = A \cup B$ 且 $A \cap B = \emptyset$），存在一个二元合并算子 $\oplus$ 满足：

$$f(D) = f(A \cup B) = f(A) \oplus f(B)$$

代数结合律是"流切片"（Stream Slicing）和"预聚合"（Pre-Aggregation）的数学基石。只有在聚合函数满足结合律的前提下，流计算系统才能够将一个长时间跨度的重叠窗口切割成物理上互不重叠的"切片"（Slices），并在切片内部对事件进行预聚合（生成切片局部的聚合结果或累加器），最后通过 $\oplus$ 算子将相关切片的中间聚合结果组合成最终的窗口值。大部分常见聚合函数（如 SUM、COUNT、MIN、MAX）都属于结合律函数。

### 可逆性（Invertibility）与回撤（Retraction）

在满足结合律的函数中，如果存在一个逆算子 $\ominus$，使得对于集合 $A$ 和它的子集 $B$，其聚合值满足：

$$f(A \setminus B) = f(A) \ominus f(B)$$

则称该聚合函数是可逆的。典型的可逆聚合包括 SUM（其逆算子为减法）和 COUNT（其逆算子为递减）。可逆性直接决定了系统能否采用超高效的"移入-移出"（Subtract-on-Evict）滑动窗口管理策略。对于可逆函数，窗口在沿时间轴滑动时，系统无需构建任何复杂的树状索引或切片聚合链。系统只需要在内存中维护一个全局的窗口累加器：当新切片滑入窗口时，执行全局累加器与新切片中间结果的 $\oplus$ 操作；而当旧切片滑出窗口时，直接执行全局累加器与失效切片结果的 $\ominus$ 操作。该更新过程具有 $\mathcal{O}(1)$ 的平摊时间复杂度，且状态复杂度极低，是滑动窗口最理想的物理实现。

### 结合但不可逆函数（Non-Invertible Associative Functions）

诸如 MIN 和 MAX 这样的函数，虽然满足结合律，但不可逆。一旦窗口内当前的最大值由于窗口滑动而失效（Evicted），系统无法通过简单的逆操作扣除该最大值，因为系统无从得知扣除后剩余元素的次大值是多少。为了在不需要缓存所有原始事件的情况下高效处理这类不可逆结合函数，系统必须退而求其次，采用分层树状组织结构（如 Segment Tree、FlatFAT 或特定的非分支扫描算法）来存储各切片的局部聚合值。当发生窗口滑动和数据失效时，系统通过 $\mathcal{O}(\log N)$ 的复杂度重新计算树中相关节点的值，从而得出新的窗口聚合结果，这避免了 $\mathcal{O}(N)$ 的全局重新扫描。

### 整体性/全量函数（Holistic Functions）

整体性函数（如 MEDIAN、百分位数 PERCENTILE 等）既不满足结合律，也不满足可逆性。此类函数无法通过任何切片中间聚合状态进行推导计算，因为两个子集的中间值或百分位数无法通过代数运算组合出并集的对应指标。对于整体性函数，任何形式的"切片预聚合"都将彻底失效。窗口管理机制必须退化为最原始的"全量缓存"模式，即系统必须在物理状态中完整保存落在当前活跃窗口内的所有原始流事件。当窗口滑动或触发计算时，系统必须对窗口内的全部数据集执行完全的重新扫描与排序（对于中位数，计算复杂度为 $\mathcal{O}(N \log N)$），从而导致极高的内存开销与严重的计算延迟。

### 聚合函数分类对比

| 聚合函数分类 | 典型函数 | 代数属性 | 窗口切片预聚合可行性 | 最佳滑动窗口更新策略 | 状态复杂度 (每 Key) | 触发时时间复杂度 |
| --- | --- | --- | --- | --- | --- | --- |
| 可逆结合函数 | SUM, COUNT, AVG (解耦为 Sum & Count) | 结合律 + 可逆性 | 可行，高度压缩（仅存中间累加器） | 移入-移出（Subtract-on-Evict）/ 环形缓冲区 | $\mathcal{O}(1)$ | $\mathcal{O}(1)$ |
| 不可逆结合函数 | MIN, MAX | 结合律，无逆操作 | 可行，高度压缩（仅存切片聚合值） | 树状层次聚合 (FlatFAT / 串行两路扫描) | $\mathcal{O}(N_{\text{slices}})$ | $\mathcal{O}(\log N_{\text{slices}})$ 或平摊 $\mathcal{O}(1)$ |
| 整体性/非结合函数 | MEDIAN, PERCENTILE | 无结合律，无逆操作 | 不可行（必须物理保留所有原始元组） | 全量缓存队列扫描 | $\mathcal{O}(N_{\text{tuples}})$ | $\mathcal{O}(N_{\text{tuples}} \log N_{\text{tuples}})$ |

## 3. 学术界高效计算方法：从流切片到层次树

针对重叠窗口中存在的计算与存储冗余，学术界提出了多种创新性的优化方案，主要围绕"流切片"（Stream Slicing）和"分层增量树"两大方向展开。

### 流切片机制与 Scotty 框架

学术界解决滑动窗口冗余的核心思想是流切片（Stream Slicing）。其基本原理是将无界数据流沿时间轴剖分为一系列互不重叠、边界确定且有限的切片（Slices）。系统为每个切片维护一个局部的预聚合值。当一个或多个重叠窗口触发计算时，系统通过将落入这些窗口边界内的所有切片的局部聚合值进行快速合并，从而输出各个窗口的聚合结果。

柏林工业大学等学术机构提出的 **Scotty** 是流切片技术的集大成者。作为一个通用且高效的窗口聚合算子框架，Scotty 实现了流切片、预聚合和多窗口/多查询间的聚合共享。Scotty 的架构包含两个核心管理组件：

- **流切片器（Stream Slicer）**：负责在流数据到达时，将其分派到对应的唯一切片中，并以增量方式更新切片状态。切片的划分基于所有活跃窗口的启动边界与结束边界。
- **切片管理器（Slice Manager）**：动态监控并维护切片的生命周期，管理切片的合并（Merging）、分裂（Splitting）和更新（Updating）。

切片划分与窗口边界对齐示例：

```
时间轴:   |---- Slice 1 ----|---- Slice 2 ----|---- Slice 3 ----|---- Slice 4 ----|
窗口 A:   [===================================] (由 Slice 1 和 Slice 2 拼接而成)
窗口 B:                     [===================================] (由 Slice 2 和 Slice 3 拼接而成)
窗口 C:                                       [===================================]
```

对于顺序流入的数据，切片的终结是单调递增的，因此历史切片一经闭合便不可修改，状态极为稳定。然而，在真实流处理中，乱序（Out-of-Order）事件非常普遍。当迟到元组（Late Tuples）到达时，它们可能会落入已经终结的历史切片中。为此，Scotty 的切片管理器支持对历史切片进行原地更新（Update），并在延迟容忍范围内重新合并输出。

在乱序流环境下，切片划分的复杂性显著增加。对于纯顺序流，切片只需在窗口开始时创建，因此切片数量至多与逻辑窗口数量一致。而对于乱序流，为了允许历史切片在任意边界上接纳迟到元组，切片边界必须在窗口的开始和结束双向对齐。这意味着乱序流切片最多会产生两倍于顺序流的切片数量，但在窗口边界重合时，实际切片数会大为减少。

### 动态会话窗口切片与状态质变

相较于边界完全确定的滑动与滚动窗口，会话窗口（Session Window）的生命周期和边界是不确定的，完全由数据流中事件之间的时间不活动间隔（Gap）决定。这使得针对会话窗口的切片管理异常复杂。学术界通过 Scotty 首次实现了面向会话窗口的高效流切片与聚合共享。Scotty 处理会话窗口的核心发现包括：

- **最小间距支配律（Smallest Gap Dominance）**：当系统同时运行多个不同会话间隔（如 $g_1 < g_2 < \dots < g_k$）的会话查询时，切片的划分逻辑由定义了最小会话间隔的查询（即 $g_1$）绝对主导。所有具有更大间隔的会话窗口结果，在物理上均可表示为这些基于 $g_1$ 划分的基础切片的有机组合（Composition）。
- **乱序状态突变管理**：由于会话边界的动态性，一个迟到乱序元组的加入不仅会更新局部切片，还可能导致会话状态发生"物理解构与重组"。Scotty 的切片管理器将其抽象为三类状态质变过程：
  - **延伸（Extension）**：迟到元组落入会话边缘的 Gap 内，导致原会话窗口在时间轴上向前或向后延伸。
  - **融合（Fusion）**：迟到元组恰好落入两个原本分离的会话窗口之间，使得两者之间的空闲时间缩短至阈值以下，从而将两个独立的旧会话以及新元组强行融合成一个超长的单会话。此时切片管理器必须将相对应的物理切片进行 Merge 操作。
  - **创建（Creation）**：迟到元组孤立地出现在远离任何活跃会话的时间段，从而激发出一个全新的独立会话。这需要切片管理器对现有的物理切片进行 Split 操作。而执行 Split 是一项昂贵的物理操作，因为它要求系统必须从内存中找回该切片范围内曾被缓存的所有原始元组，以重构两个新切片的局部累加器，这也迫使系统在此时必须保留部分原始事件拷贝。

### 层次聚合树：FlatFAT 与 Reactive Aggregator

除了流切片外，如何对大量切片的预聚合值进行高效合并是另一个研究重点。在处理不可逆的结合函数（如 MIN/MAX）时，若直接进行线性扫描合并，时间复杂度仍会随着窗口内切片数量 $N$ 的增加而退化为 $\mathcal{O}(N)$。为了实现高吞吐和低延迟，学术界在 Reactive Aggregator (RA) 框架中提出了 **FlatFAT** (Flat Fixed-size Aggregator Tree)。

FlatFAT 摒弃了传统的、依赖大量内存指针构建的二叉树（如常规线段树），转而采用连续的扁平数组来物理存储整棵二叉聚合树。树的叶子节点对应于各个时间切片的局部聚合累加器，而树的非叶子内部节点则代表其子节点通过二元结合算子 $\oplus$ 合并后的中间状态。当新元组滑入导致某个切片更新，或者旧切片过期被驱逐时，FlatFAT 能够在连续的内存空间内以 $\mathcal{O}(\log N)$ 的极低耗时完成自底向上的节点值传播与更新。当窗口触发查询时，系统只需在 FlatFAT 的数组结构上进行几步非指针跳转的算术定位，即可将窗口所覆盖的若干子树根节点值进行结合，瞬间产生聚合结果，从而在硬件层面极大地提升了缓存友好性，避免了内存碎片。

### 学术界主流优化方案的技术演进

学术界在滑动窗口优化方面进行过长达二十年的探索，不同方案在乱序支持、内存占用以及支持的聚合函数类型上呈现出明显的演进脉络。

- **Panes 与 Pairs 机制**：早期的 Panes 机制尝试将窗口细分为固定大小的子窗格进行聚合，但其主要针对严格有序的数据流，且仅对简单的滑动窗口有效，不具备对乱序及会话窗口的适应力。Pairs 机制在此基础上对相邻窗口边界进行了配对，但在应对复杂组合查询时状态开销仍显冗余。
- **B-INT 树状索引**：提出了基于树状结构的增量聚合思想，但其结构设计较为笨重，在面临大规模迟到元组更新历史节点时，树结构调整的物理开销（Update Complexity）过高。
- **SABER、Cutty 与 Fragments**：SABER 将流切片用于 GPGPU 计算以追求极致吞吐，但其强制要求数据流完全有序，无法应对现实中的延迟和乱序。Cutty 专注于解决用户自定义窗口（UDW），但由于缺乏对 out-of-order 状态的弹性容错，其实用性受限。Fragments 则在单查询切片上有较好表现，但不支持跨会话窗口的多查询聚合共享。
- **Scotty 与 FlatFAT 的现代结合**：Scotty 通过抽象出 Slice Manager，统一了对乱序流、多窗口组合（Sliding、Tumbling、Session）的切片划分和聚合共享。而 FlatFAT 则在底层算法上解决了结合非逆函数在切片合并时的效率瓶颈，两者相辅相成，代表了当前学术界窗口聚合的最佳实践。

## 4. 前沿工业界流处理系统的窗口实现与管理实践

工业界系统在吸收学术界理论成果的同时，更加注重系统的高可用性、状态的持久化、一致性保证以及在大规模分布式集群下的可扩展性。不同的工业系统基于其自身的架构定位，衍生出了各具特色的窗口实现路径。

### 4.1 Apache Flink 窗口架构演进：双轨制演进

Apache Flink 在工业界以其强大的状态管理机制和精确一次（Exactly-Once）保证而著称。针对窗口聚合，Flink 采用了双轨制的设计演进路线：DataStream API 采用基于物理窗格（Pane）的常规实现，而 SQL/Table API 则引入了基于切片（Slice）的先进 TVF 架构。

#### Flink DataStream API 的 Pane-Based 窗口管理

在 DataStream API 中，当用户调用 `.keyBy().window()` 时，Flink 的物理算子会为每个 Key 创建对应的窗口触发器（Trigger）和状态存储。Flink 在分布式状态中，本质上将每个逻辑窗口独立表示为一个物理键值存储条目（Key-Window Pair）。当面对滑动窗口时，Flink 默认将每个到来的流元素复制并分发到所有关联的逻辑窗口中。虽然用户可以使用 `ReduceFunction` 或 `AggregateFunction` 来保持局部状态的大小（即状态中仅存储当前的 Accumulator 而非原始元组列表），但在物理状态后端中，Flink 仍必须为每一个重叠的 Window 维护一个独立的 Key-Value 记录。当滑动步长极小时，活跃窗口数量激增，状态后端（尤其是 RocksDB）将面临严重的写入放大和存储膨胀。

在大规模流作业中，使用 RocksDB 作为状态后端时，为了抗住高频滑动窗口带来的吞吐压力，必须进行深度的状态底层调优：

- 调整 `block_cache_size`：增大未压缩数据块（Uncompressed Blocks）的物理缓存，确保在频繁更新和读取窗口状态（Accumulator）时尽可能命中内存，避免频繁触发 SSTable 的磁盘 I/O 读操作。
- 优化 `write_buffer_size` 与 `write_buffer_number`：显著调大 Memtable 的内存空间和缓冲区个数，以便将频繁的窗口增量更新合并缓冲在内存中，延缓其向磁盘刷写 SSTable 的频率。
- 设置 `min_write_buffer_number_to_merge`：该参数控制在向磁盘 Flush 之前需要合并的最小 Memtable 数量，建议设置为 2。通过在内存中提前合并针对相同 Key-Window 的频繁修改版本，可以大幅度减少落地到 SSTable 磁盘文件中的冗余历史记录。

此外，在实际应用中，Flink 默认的 Tumble 或 Slide 窗口会在全分区（Partitions）范围内同时到达结束时间，这会导致全网所有 TaskManager 上的窗格在同一瞬间触发 Trigger 计算并向外发送结果，对下游 Cassandra 或 Kafka 等系统产生剧烈的脉冲式写入洪峰，随后系统又进入闲置状态。为了避免这种群集效应，工业界常通过继承 `ProcessFunction` 实现自定义低延迟窗口。该自定义窗口采用事件首次到达时加特定偏移来对齐，从而使得各 Partition 上的窗格自然交错（Staggered）关闭，实现下游流量的平滑消峰。

#### Flink SQL & Table API 的 Slice-Based 窗口优化

为了彻底解决 DataStream API 的状态膨胀和数据复制问题，Flink 在 SQL/Table 引擎中完全重构了窗口算子，推行了基于窗口表值函数（Windowing TVF）的切片式窗口（Slice-Based Windowing）。在 Flink SQL TVF 模式下（如 TUMBLE、HOP、CUMULATE），SQL 优化器在生成物理执行计划时，会自动将滑动窗口（HOP）剖分为一系列逻辑上不重叠的时间切片（Time Slices）。物理算子内部不再为每个重叠的滑动窗口各自维护状态累加器，而是仅仅为当前活跃的非重叠 Slice 维护一个唯一的局部聚合状态。当事件到达时，Flink 将其分配到唯一的 Slice 中进行本地增量预聚合。只有当某个切片的时间跨度终结，或者窗口通过 Watermark 判定闭合时，Flink 才会调用内置的聚合 Merge 函数，将关联切片的局部聚合值融合成最终的 SQL Row 发送给下游，并随即清理已经失效的历史切片状态。这一机制不仅将状态大小削减了数倍，还使得内存利用效率达到了学术界流切片的同等水平。

### 4.2 Apache Beam 与 Google Cloud Dataflow：统一的回撤流模型

Apache Beam 及其核心云端执行引擎 Google Cloud Dataflow 是 Dataflow 流式处理模型的鼻祖与典范。在 Beam 的编程范式中，无界数据集（PCollection）通过 Windowing 机制切分成有限的视口。在大规模滑动窗口计算上，Beam 默认仍然遵循传统的物理元素多分法复制，每个元素被标记并路由到其所属的各个重叠窗口内，存在与 Flink DataStream 类似的开销权衡。但 Beam 的独特优势在于它将时间、完整性与聚合输出进行了极致的解耦，引入了高度抽象的乱序容错架构，具体由四个核心维度定义：

- **Watermarks（何时算完）**：代表系统在事件时间轴上推进的进度度量，用于判定窗口是否可以安全闭合。
- **Allowed Lateness（能等多久）**：允许用户指定一个时间窗口，在此期间即使 Watermark 已经跨过窗口结束时间，算子仍需保留窗口的物理状态，以便随时接纳迟到的数据。
- **Triggers（何时输出）**：定义了何时将窗口聚合的结果真正向下游发射，支持"早于 Watermark（Early Firings）"、"Watermark 到达（On Time）"以及"晚于 Watermark（Late Firings）"的多阶段触发机制。
- **Accumulation & Retraction 模式（如何修正）**：当迟到元组在 Allowed Lateness 内到达并触发 Late Firings 时，Beam 提供了"累积（Accumulating）"和"回撤（Retracting）"两种输出策略。在回撤模式（Retracting Mode）下，当窗口的聚合结果发生变化时，Beam 算子会向后方算子发送一个携带先前旧结果的回撤（撤销）信号（如负增量），随后紧跟一个全新的纠正后的聚合行，这构成了高容错流式管道的核心。

### 4.3 Materialize：基于微分数据流的虚拟时间窗

Materialize 是一款独特的 SQL 流式数据库，其底层引擎基于微软研究院开发的 Timely Dataflow 分布式数据流计算框架以及 Differential Dataflow 微分增量计算框架。Materialize 在窗口聚合上的实现逻辑与前述所有物理时间窗口系统有着本质的不同。

#### 微分增量计算原理

在 Materialize 中，所有的数据集和流都被统一抽象为支持正负多重性（Multiplicities，即频数）的多重集（Multisets）。任何源端的变化，无论是插入、修改还是删除，在底层都表示为一元或多元微分更新流，其基础数据格式为：

$$\text{Update} = (\text{Data}, \text{Time}, \text{Diff})$$

其中 Time 是由 Epoch 和迭代步数组成的多维逻辑时间戳，而 Diff 是一个可以为正或为负的整型数（例如，插入一条新订单为 $+1$，回撤或删除旧记录为 $-1$）。

#### 虚拟时间窗与主动回撤注入

Materialize 不需要在物理上划分和运行任何类似于 Flink 或 Beam 内部的"物理时间窗口算子"。它对于滑动窗口的处理，采用了一种极具创意的时间差分回撤主动注入机制。例如，当一条新的交易事件 $d$ 在源端产生，其携带的时间戳为 $t$（假设滑动窗口大小为 $W$）。Materialize 接收到该事件后，会在流计算图的始端主动生成并向后方流管道中注入两条互成镜像的微分更新记录：

- 一条立即生效的插入差分：$(d, \text{Timestamp}(t), +1)$
- 一条在未来 $t + W$ 时刻自动生效的、具有相同 Key 的物理回撤差分：$(d, \text{Timestamp}(t + W), -1)$

当这两条记录流入底层的增量 reduce 聚合算子时，reduce 仅仅需要在内部的 KV 状态中维护当前的聚合值差分。在逻辑时间到达 $t$ 时，算子增量累加数据；当逻辑时间单调递增跨过 $t + W$ 时，未来回撤更新 $d(t+W, -1)$ 自动生效，触发聚合算子增量地执行减除操作，从而完成了完美的滑动窗口生命周期闭环。

```
输入事件到达 (时间 t)  ====>  注入 (Data, t, +1)       ===> 增量累加聚合
                        ====>  注入 (Data, t + W, -1)   ===> 延迟到 t+W 自动触发回撤
```

#### 极简的状态存储（Arrangements）

由于采用了全差分设计，Materialize 能够将流计算的内部状态高度压缩，并提炼为一种被称为 Arrangement 的 LSM-Tree 变体索引结构。Arrangement 本质上是一个持久化的 Key -> (Data, Time, Diff) 的映射图。Materialize 在设计上通过前置的字段投影和预聚合（Pre-reduction），在微分注入阶段将不必要的元数据列剔除，将每条状态记录压缩至 $0 \sim 16$ 字节（相比之下，传统实现的记录往往高达 $96$ 字节），并在多路独立创建的 SQL Materialized Views 之间共享同一个 Arrangement 物理索引，从而在大规模滑动聚合中实现了极致的内存复用。

### 4.4 RisingWave：云原生流数据库的 OverWindow 与 EOWC 机制

RisingWave 是一款专为云原生时代构建的高性能流式数据库，采用存储与计算完全分离的现代架构。其内部核心状态均保存在名为 Hummock 的分布式行式 LSM-Tree 存储引擎中，该引擎的数据块持久化存储于低成本的云端对象存储（如 AWS S3）中，计算节点仅在本地内存及高速 NVMe 盘上保留高频热数据的 Read Cache。

#### OverWindow 算子的双阶段流水线设计

RisingWave 在其 1.1 版本中重磅推出了针对 Over Aggregation（在 SQL 中表现为带有 OVER 子句的窗口函数）的 OverWindow 物理执行算子。其内部窗口滑动计算采用了优雅的双阶段流水线设计：

```
数据源流入 ===> --(对齐与排序)--> ===> 输出
```

- **第一阶段：SortBuffer**：由于流计算中 Watermark 具有非递减的单调属性，SortBuffer 作为一个高内存效率的缓冲区，专门接收非确定性乱序到达的数据行。它利用 Watermark 作为时钟，仅在判定当前数据行前的历史排序已经完全确定（即 Condition ① 满足，ORDER BY 字段的值已被 Watermark 浸没覆盖）时，才按排好的严格时间顺序向第二阶段释放数据。
- **第二阶段：SlidingWindow 状态机**：该阶段专门接收 SortBuffer 输出的、具有严格时序和 Partition 分区的有序行流。在状态机中，OverWindow 算子为每个 Partition 分区维护一个当前正在等待视口闭合的行游标（CURRENT ROW）以及与之对应的滑动物理视口范围（CURRENT WINDOW）。当新的一行落入视口导致窗口完整度改变（Condition ② 满足，即窗口的最后边界行也已被确定）时，SlidingWindow 状态机立即计算最终结果并将该行向外发射，随后控制 CURRENT ROW 游标沿严格的逻辑序列向下滑动。随着窗口的滑动，那些已经彻底跌出视口左侧范围的古老历史行会被 OverWindow 算子从 Hummock LSM 状态表中安全且无残留地主动清除（Purge），从而实现了完美的内存控制。

#### Emit 策略与 EOWC 机制的性能跃升

作为一个旨在通过 Materialized Views 提供即时低延迟查询的数据库系统，RisingWave 支持两种截然不同的输出与状态生命周期策略：

- **Emit-on-Update 模式**：任何微小的数据变动或回撤，都会导致 OverWindow 算子即时重新计算受影响范围内所有相关行的最新结果，并向下游和物化视图写入增量更新。这提供了亚秒级的视图新鲜度，但频繁的增量维护和历史状态检索给底层 Hummock LSM 存储带来了巨大的 Read/Write 放大效应。
- **Emit-on-Window-Close (EOWC) 模式**：当用户在 SQL 物化视图定义末尾追加 `EMIT ON WINDOW CLOSE` 关键字时，系统将开启 EOWC 计算模式。在此模式下，算子在接收到闭合特定窗口的 Watermark 之前，会保持静默，不在视图中产生中间更新，不产生撤销和回撤元组。当 Watermark 最终断定该窗口已闭合后，RisingWave 才会单次发射该窗口的追加（Append-Only）聚合结果，并在物化视图中将其固化，随即在底层状态表中彻底物理抹除该闭合窗口下的所有历史缓存。对于 OVER 窗口、分位数聚合等操作，EOWC 能够让 RisingWave 在内部启用高度集中的批处理算法，完全避免了为了支持后续可能更新而进行的中间状态维护。这极大地削减了 Hummock 状态存储的吞吐负担，使得吞吐性能和空间释放效率相比于增量更新模式呈现出数量级的跃升。

## 5. 工业界与学术界主流系统及技术的深度对比

| 维度 | Flink DataStream API | Flink SQL (Table API) | Apache Beam / Dataflow | Materialize | RisingWave | Scotty (学术界) |
| --- | --- | --- | --- | --- | --- | --- |

## 6. 结论与架构选型建议

流计算中窗口聚合算子的演进历程，本质上是一场在内存空间开销、计算时间复杂度和窗口语义普适性三者之间寻找最优解的代数与架构技术革命：

- **学术界的演进启示**：学术界证明了，在聚合函数满足代数结合律的前提下，通过将重叠的时间窗口物理投影为非重叠的时间切片（Stream Slicing），能够在逻辑上完美剥离滑动窗口带来的状态膨胀问题。而对于结合但不可逆函数（如 MIN/MAX），采用以 FlatFAT 为代表的树状层次聚合结构是绕过 $\mathcal{O}(N)$ 扫描瓶颈的最优雅学术路径。
- **工业界的流派分裂与收敛**：
  - Apache Flink 为代表的传统流处理器，正在从早期物理冗余巨大的 Pane 窗格复制架构（DataStream API），向高内存利用率的 Slice 物理切片架构（SQL TVF）全面收敛。对于超大滑动窗口，若仍使用 DataStream，必须对 RocksDB 的 Memtable 融合与 Write Buffer 进行深度性能压榨。
  - Materialize 凭借差分数据流引擎，走出了"无物理窗口"的极简主义道路。通过在逻辑时间戳轴上主动对冲注入未来的"回撤"元组，将滑动窗口管理转变为高度一致且普适的增量求和。此方式在大规模多视图 Arrangement 共享上优势无可比拟，但其底层的逻辑时钟追踪也引入了更高的系统复杂性。
  - RisingWave 代表了存算分离时代的最高水准，其独特的双阶段 OverWindow（SortBuffer 彻底隔离乱序，SlidingWindow 状态机纯粹单向滑动）提供了高度确定的流控保障。同时，EOWC 模式的引入彻底摆脱了中间回撤波动的吞吐损耗，是下游单向追加系统（如 Kafka/S3）的黄金搭档。

### 针对系统架构师的选型及优化建议

- **若业务充斥着超高滑动比（High Overlap Factor）的结合型指标**：推荐优先选用 Flink SQL (Table API) TVF 或 RisingWave，它们在物理层面实现了切片式状态存储，相比 Flink DataStream API 能节省 90% 以上的内存和 I/O 开销。
- **若业务需要对历史数据进行高频且精准的撤销、回溯与订正**：应当首选 Materialize 或基于 Apache Beam (Dataflow) 的 Retraction 模型。Materialize 强大的差分回撤注入天然适应任何维度和深度的撤销逻辑。
- **若集群运行在多租户、需要弹性伸缩的云端环境且窗口状态巨大**：推荐选用 RisingWave。其存算分离的 Hummock 引擎能将数 TB 的窗口状态直接卸载到低成本的对象存储中，避免了 Flink RocksDB 本地磁盘写满的物理隐患，并且能实现近乎瞬间的故障恢复与节点弹性拉起。
